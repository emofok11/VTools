## UI 一致性通用规则

### RULE-001 同类型按钮间距一致

- 同一层级、同一用途的按钮，`gap`、高度、左右内边距应保持一致
- 不要只看数值，要看视觉结果
- 如果视觉不一致，优先检查按钮自身 `padding`，不要只改外层 `gap`

### RULE-002 同类型容器内容一致

- 同类型容器中的文字大小、字重、颜色、间距应保持一致
- 标题与版本号这类相邻元素，不仅要看外层排列，还要看各自内部留白
- 同一套视觉系统内，矩形标签应避免一边紧、一边空的情况

### RULE-003 输入框或底板四边留白一致

- 输入框、标签底板、chip、badge 这类容器，内部上下左右留白应平衡
- 若文本较短，固定宽度容易让左右留白看起来过大，需要优先检查 `width`
- `border-radius` 与 `padding` 要一起评估，不能只改单一属性

### RULE-004 先对照正确基线，再修问题

- 先找到项目里已经正确的同类组件，再决定如何修改
- 优先复用正确组件的 `padding`、`gap`、`border-radius`、`width` 规则
- 不要为了修一个局部问题，顺手重做整块结构

### RULE-005 最小改动原则

- 优先改局部选择器，避免全局联动
- 优先改 `padding`、`gap`、`border-radius`、`width`
- 结构、DOM 层级、公共样式只有在必要时才动
- 修复新问题时，不能破坏已有已修好的功能

### RULE-006 展示态与编辑态必须一起检查

- 展示态正确，不代表编辑态正确
- 版本号、标题、输入框这类元素，要同时检查：
  - 展示态
  - 编辑态
  - hover 态
  - 聚焦态
- 常见问题是展示态改对了，但编辑态仍保留旧的 `padding` 或旧的 `width`

### RULE-007 视觉间距不等于骨架间距

- 外层 `gap` 一样，视觉效果仍可能不一致
- 根因通常来自：
  - 元素自身 `padding`
  - 元素自身 `border-radius`
  - 固定 `width`
  - 字重或字号差异
- 所以排查顺序应为：先元素自身，再外层容器

### RULE-008 预览文档属于独立视觉体系

- `DocumentPreview` 的版头不能直接套用编辑器 header 的统一规则
- 预览页更接近最终出稿版式，优先保留原稿节奏
- 如果预览文档被误改，应优先定点回滚，不要顺手同步其它体系

### 高频误区

- `gap` 一致，就误以为视觉间距一致
- 只改了 `border-radius`，但 `padding` 仍然让版本号看起来过宽
- 只改展示态，漏掉编辑态
- 为了统一 header，把预览文档也一起改掉
- 没有先找基线，直接在多个文件里凭感觉同步

### 修改前检查清单

- 当前问题属于编辑器体系，还是预览文档体系
- 项目里是否已经有一个正确的对照组件
- 问题根因更像是 `padding`、`gap`、`border-radius` 还是 `width`
- 是否需要同时调整展示态和编辑态
- 是否存在误伤已经修好的区域的风险

### 修改后检查清单

- 返回按钮、标题、版本号三者的视觉节奏是否稳定
- 同类按钮或容器是否仍保持一致
- 文本四周留白是否均衡
- 预览态和编辑态是否都正常
- 是否只影响了目标区域，没有误改其它页面
